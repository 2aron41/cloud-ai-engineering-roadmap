## How I Fixed Them

- Deleted the incorrectly named `index-.html` object.
- Uploaded the correct file as exactly `index.html`.
- Verified its new size and modification time in S3.
- Confirmed the file contained `Live on AWS`.
- Used `/*` to clear all cached CloudFront paths.
- Waited until the invalidation status was `Completed`.
- Copied the exact distribution domain from the CloudFront console.
- Tested the live website in a new private browser window.

## What I Learned

- Replacing a file in S3 does not always immediately update the CloudFront website because cached copies may remain.
- An invalidation removes selected files from CloudFront’s cache.
- `/index.html` targets one path, while `/*` clears all cached paths.
- CloudFront’s default root object must match the exact S3 object key.
- Small filename differences create separate S3 objects.
- The exact CloudFront distribution domain should be copied from the console rather than typed manually.
- GitHub should remain the source of truth for website code.
- The manual deployment workflow should eventually be replaced by GitHub Actions.
